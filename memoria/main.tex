\renewcommand{\abstractname}{Resumen}
\begin{abstract}
\setlength{\parindent}{0pt}
\setlength{\parskip}{0.6em}

En este trabajo se presenta el desarrollo de una prueba de concepto de un agente con capacidad de razonamiento basado en modelos de lenguaje de gran tamaño \emph{(LLMs)}.
A diferencia de los sistemas tradicionales que generan una respuesta directa ante una consulta, el agente propuesto implementa bucles dinámicos de \emph{Chain of Thought (CoT)} que le permiten generar,
evaluar y refinar pasos intermedios antes de producir una respuesta final.

El sistema se implementa utilizando modelos de lenguaje ejecutados en local, como Llama-3, junto con la librería \emph{LangGraph} para la contrucción de flujos de razonamiento. Además, se emplea
una estrategia de búsqueda basada en \emph{Beam Search} que permite explorar múltiples posibles líneas de razonamiento y seleccionar aquellas más prometedoras mediante un mecanismo de puntuación.

El objetivo del proyecto es analizar la viabilidad de este enfoque para mejorar la capacidad de razonamiento de agentes basados en LLMs y estudiar su comportamiento en diferentes escenarios de
resolución de problemas.
\medskip

\noindent \textbf{Palabras clave:} modelos de lenguaje, agentes inteligentes, \emph{Chain of Thought}, razonamiento automático, \emph{Beam Search}.
\end{abstract}


\renewcommand{\abstractname}{Abstract}
\begin{abstract}
\setlength{\parindent}{0pt}
\setlength{\parskip}{0.6em}

This work represents the development of a proof-of-concept reasoning agent based on Large Language Models \emph{(LLMs)}.
Unlike traditional systems that generate an immediate response to a query, the proposed agent implements dynamic \emph{Chain Of Thought (CoT)} loops that allow it to generate, evaluate, and refine
intermediate reasoning steps before producing a final answer.

The system is implemented using locally executed language models such as Llama-3, together with the \emph{LangGraph} library for building reasoning workflows. Additionally, a \emph{Beam Search}
strategy is employed to explore multiple possible reasoning paths and select the most promising ones through a scoring mechanism.

The goal of this project is to analyze the feasibility of this approach to improve the reasoning capabilities of LLM-based agents and to study their behaviour in different problem-solving scenarios.
\medskip

\noindent \textbf{Keywords:} language models, intelligent agents, \emph{Chain of Thought}, automated reasoning, \emph{Beam Search}.
\end{abstract}

\chapter{Introducción}
\section{Motivación}
%\setlength{\parindent}{0pt}
%\setlength{\parskip}{0.5em}

En los últimos años, el desarrollo de los \emph{Large Language Models (LLMs)} ha supuesto un avance significativo en tareas de procesamiento
del lenguaje natural. Sin embargo, a pesar de la capacidad para generar texto coherente y resolver una amplia variedad de tareas, estos modelos
presentan limitaciones en lo que a enfrentar problemas que requieren un razonamiento complejo se refiere, ya que tienden a generar respuestas
directas sin una estructura intermedia explícita.

La ausencia de pasos intermedios limita la capacidad de valorar el razonamiento, desencadenando en falta de transparencia y control sobre cómo se
construye el razonamiento.
Es por ello que surge la necesidad de desarrollar sistemas que no solo generen respuestas, sino que estructuren su proceso de razonamiento y evalúen
sus propios pasos intermedios, con un enfoque que facilite la experimentación y análisis del comportamiento del modelo.

En conclusión, este proyecto se plantea a partir de la motivación por construir algo más complejo que una simple llamada a un modelo, profundizando
así en un área en crecimiento, desde el interés por entender no solo qué responden estos modelos, sino cómo llegan a hacerlo.

\section{Trabajos previos}

Con el fin de abordar la limitación que presentan los \emph{Large Language Models (LLMs)} anteriormente mencionada, diversos trabajos han propuesto
técnicas que fomentan el razonamiento paso a paso. Un ejemplo de esto es \emph{"Chain-of-Though Prompting (CoT)"}, introducido por Wei et al. (2022) [1],
el cual consiste en guiar al modelo para que genere pasos intermedios antes de alcanzar una respuesta final. Este enfoque pone de manifiesto la importancia
de estructurar el proceso de generación en lugar de limitarse a obtener una respuesta final directa.

Por otro lado, en el ámbito de la generación de texto, los algoritmos de búsqueda han sido ampliamente utilizados para explorar múltiples
posibles soluciones. Destaca el algoritmo de \emph{Beam Search}, el cual se apoya en una función de evaluación para mantener las secuencias más
prometedoras en cada iteración. Trabajos como el de Wu et al. (2020) [3] demuestran que, aunque este método haya sido tradicionalmente aplicado en tareas como la traducción automática,
su uso puede extenderse también al razonamiento en \emph{LLMs}, simplemente permitiendo explorar distintas cadenas de pensamiento
\emph{(CoT)} y seleccionando las más relevantes (según la función de evaluación anteriormente mencionada).

Estos antecedentes reflejan dos ideas fundamentales:
\begin{itemize}
  \item Por un lado, la necesidad de guiar el razonamiento de los modelos mediante la generación de pasos intermedios.
  \item Por el otro, la utilidad de explorar múltiples soluciones antes de seleccionar la respuesta final.
\end{itemize}

Sin embargo, la combinación de ambos enfoques en sistemas que implementen bucles dinámicos de razonamiento y mecanismos de selección de hipótesis
continua siendo un área en pleno desarrollo.

\section{Objetivos}

El objetivo principal de este trabajo es desarrollar una prueba de concepto de un agente basado en modelos de lenguaje capaz de incorporar un proceso de
razonamiento estructurado, en lugar de generar respuestas de forma directa. De esta forma, y con el objetivo de facilitar un aálisis detallado del proceso
seguido por el modelo, se busca explorar un enfoque que permita descomponer la resolución de un problema de índole lingüística en una serie de pasos intermedios.

De forma más concreta, se pretende implementar un sistema que no dependa únicamente de un camino de razonamiento, sino que sea capaz de generar múltiples posibles
trayectorias, permitiendo así analizar y estudiar como evoluciona el proceso de generación de cada una de ellas. Asimismo, un mecanismo evaluará la calidad de dichos
razonamientos generados, con el fin de seleccionar aquellos que resulten más adecuados con el tipo de problema que con el que se está trabajando. Este proceso de
evaluación se utilizará por tanto para guiar la exploración del espacio de soluciones, descartando las opciones menos prometedoras.

Otro gran objetivo de este proyecto se centra en la estructuración del funcionamiento del sistema mediante un flujo de ejecución organizado, que facilite no solo la
comprensión de este, sino también su posible extensión en trabajos futuros.

Por último, se busca analizar el comportamiento del agente desarrollado a través de problemas de índole lingüística...

\section{Imaging model and objectives}
Assume \(y = A x + n\). The SR network \(f_\theta\) estimates
\(\hat x = f_\theta(y)\).
We use the composite objective
\begin{equation}
  \mathcal{L}_{\mathrm{SR}}
  = \lambda_1 \lVert \hat x - x \rVert_1
  + \lambda_s \left(1-\operatorname{SSIM}(\hat x, x)\right)
  + \lambda_{\mathrm{tv}} \operatorname{TV}(\hat x).
  \label{eq:sr_loss}
\end{equation}
For segmentation logits \(z = g_\phi(\hat x)\) and mask \(m\),
\begin{equation}
  \mathcal{L}_{\mathrm{seg}}
  = 1 - \operatorname{Dice}\!\left(\sigma(z), m\right)
    + \beta\, \operatorname{CE}\!\left(\sigma(z), m\right).
  \label{eq:seg_loss}
\end{equation}
Total loss:
\(\mathcal{L} = \mathcal{L}_{\mathrm{SR}} + \gamma\,\mathcal{L}_{\mathrm{seg}}\).

\section{Metrics}
Performance metrics
\begin{align}
\operatorname{PSNR}(x,\hat x) &= 10\log_{10}\!\frac{L^2}{\operatorname{MSE}(x,\hat x)},\\
\operatorname{SSIM}(x,\hat x) &= \frac{(2\mu_x\mu_{\hat x}+C_1)(2\sigma_{x\hat x}+C_2)}
{(\mu_x^2+\mu_{\hat x}^2+C_1)(\sigma_x^2+\sigma_{\hat x}^2+C_2)},\\
\operatorname{Dice}(p,m) &= \frac{2\sum p m}{\sum p + \sum m}.
\end{align}

\section{Introducción a LangGraph}

En los sistemas basados en modelos de lenguaje, el enfoque tradicional ha consistido en realizar llamadas independientes y aisladas a un modelo, evolucionando posteriormente
hacia sistemas más complejos capaces de gestionar flujos de ejecución estructurados. Esto cambio responde así a la necesidad de construir aplicaciones que no solo generen
respuestas, sino que integren múltiples pasos de procesamiento, toma de decisiones y manejo de información intermedia.

En este contexto surge \emph{LangGraph}, una herramienta que se presenta como alternativa a los enfoques lineales, y diseñada para facilitar la construcción de agentes
mediante la definición de flujos de ejecución basados en grafos. \emph{LangGraph} modela procesos de forma más flexible, en los que pueden existir bifurcaciones, ciclos y
dependencias entre distintos pasos. \cite{langgraph2024}

A diferencia de un flujo secuencial, un grafo permite mayor flexibilidad, al representar de manera más natural procesos complejos, en los que el sistema puede iterar,
tomar decisiones en función del estado actual o repetir determinadas operaciones hasta alcanzar un resultado adecuado. Este tipo de estructura es clave en sistemas que requieren
un comportamiento dinámico, como es el caso de agentes con capacidad de razonamiento iterativo.

De este modo, \emph{LangGraph} constituye una base sólida para el desarrollo de agentes avanzados, proporcionando un marco de trabajo que permite superar las limitaciones de los

%────────────────────── Concepto de grafos aplicado a agentes ──────────────────────────
\section{Concepto de grafo aplicado a agentes}

Los grafos han sido y son ampliamente utilizados en diferentes ámbitos de la informática para modelar sistemas complejos, ya que permite
representar de forma clara las conexiones y dependencias entre los diferentes componentes. Su estructura matemática consta de un conjunto de elementos,
denominados nodos, conectados entre sí mediante relaciones llamadas aristas. \cite{cormen2009algorithms}

En el contexto de los agentes basados en modelos de lenguage, los grafos ofrecen una forma natural de estructurar el flujo de ejecución. Dentro de este
contexto, los nodos representan etapas del proceso, mientras que las transiciones entre ellos definen el flujo de ejecución del sistema. En otras palabras,
cada nodo corresponde a una operación específica, mientras que las aristas determinan cómo se transita entre ellas.

Este enfoque permite superar ciertas limitaciones de los sistemas lineales tradicionales, en los que la ejecución sigue un único camino predefinido. Además,
los grafos facilitan la representación de procesos iterativos, en los que el sistema puede volver a ejecutar determinadas partes del flujo hasta cumplir una
condición concreta. Esta característica resulta especialmente relevante en el diseño de agentes con capacidad de razonamiento, donde es habitual generar, evaluar
y refinar soluciones de manera progresiva. \cite{bondy1976graph}

Otra ventaja a tener en cuenta con el uso de grafos es su interpretabilidad, pues su capacidad para reflejar de forma explícita la estructura
del sistema facilita actividades como el análisis del comportamiento del agente y la identificación de posibles puntos de mejora.

En consecuencia, el uso de grafos en la construcción de agentes permite diseñar sistemas más modulares, flexibles y adaptables, facilitando la implementación
de flujos de ejecución complejos que serían difíciles de representar mediante enfoques puramente secuenciales.


%────────────────────── Conceptos fundamentales de LangGraph ──────────────────────────
\section{Conceptos fundamentales de LangGraph}

Para comprender el funcionamiento de \emph{LangGraph}, es necesario analizar los elementos básicos que conforman su modelo de ejecución.
En conjunto, estos elementos proporcionan una base para el diseño y construcción de agentes complejos, permitiendo combinar generación, evaluación y toma
de decisiones dentro de un mismo sistema:

\subsection{Estado}

Un estado representa la información que se mantiene a lo largo de la ejecución del sistema. Este estado actúa como un contenedor de datos, que permite almacenar
información relevante a medida que el agente progresa, logrando así que cada paso ejecutado tenga acceso al contexto generado en las etapas anteriores.
De este modo, se facilita la construcción de aquellos procesos que dependen del historial o de resultados intermedios.

\subsection{Nodo}

Un nodo es un punto de intersección, unidad básica de procesamiento dentro del grafo. En esencia, actúa como un punto de referencia que permite organizar datos de
manera ordenada y eficiente. Cada nodo se encarga de ejecutar una operación concreta, como puede ser la generación de contenido, la evaluación de una respuesta o la
transformación de datos almacenados en el estado. Esta separación en nodos permite estructurar el sistema de forma modular, asignando responsabilidades específicas
a cada componente. Además, cuanto más rica y conectada sea la red de nodos, mayor es la capacidad del sistema para interpretar y generar conocimiento. \cite{nodes_msmk2025}

\subsection{Arista}

Las aristas o transiciones definen cómo se produce el paso de un nodo a otro. Estas transiciones pueden ser de dos tipos:
\begin{itemize}
  \item De tipo fijo, siguiendo una secuencia predeterminada.
  \item De tipo condicional, en función de determinados criterios evaluados durante la ejecución.
\end{itemize}
El uso de aristas introduce lógica de decisión dentro del flujo, adaptando el comportamiento del sistema según la información disponible en cada momento. \cite{aperezl_langgraph2024}

\subsection{Bucles}

La presencia de bucles dentro del grafo facilita la implementación de procesos iterativos gracias a que, a diferencia de los enfoques lineales,
\emph{LangGraph} permite que la ejecución vuelva a nodos previamente visitados. Esta característica resulta especialmente útil en sistemas que requieren
refinamiento progresivo, como los basados en razonamiento paso a paso.

\subsection{Puntos de inicio y finalización}

Finalmente, es necesario considerar la existencia de puntos de inicio y finalización dentro del grafo. El nodo inicial marca el punto de salida, desde donde
da comienzo la ejecución, mientras que las condiciones de finalización controlan cuándo se ha alcanzado un resultado satisfactorio y, por lo tanto,
detener el proceso. Este mecanismo permite controlar el flujo de ejecución y evitar iteraciones innecesarios.


%────────────────────── Conceptos fundamentales de LangGraph ──────────────────────────
\section{Funcionamiento general de LangGraph}

Una vez definidos los elementos fundamentales de \emph{LangGraph}, resulta necesario comprender cómo interactuan entre sí durante la ejecución de un agente. El funcionamiento de
\emph{LangGraph} puede entenderse como la evaluación progresiva de un grafo, donde el sistema transita entre los diferentes nodos siguiendo las transiciones definidas (arsitas) y
actualizando el estado en cada paso.

La ejecución comienza a partir de una configuración inicial, que contiene la información de partida necesaria para la ejecución. A partir de este punto,
se accede al nodo inicial del grafo, que actúa a su vez como punto de entrada del sistema. La modificación de estado producida por la operación ejecutada por el nodo inicial será
utilizada en las etapas posteriores.

Tras la ejecución de un nodo, el sistema evalúa todas las posibles transiciones disponibles en dicho instante.
Dependiendo de la configuración del grafo, la siguiente etapa puede:

\begin{itemize}
  \item Estar determinada de forma fija.
  \item Depender de condiciones evaluadas a partir del estado actual.
\end{itemize}

Con dicho condicionante, el mecanismo logra introducir dinamismo en el flujo de ejecución, permitiendo que el agente sea capaz de adaptarse a la información generada en tiempo
de ejecución, y adoptar el comportamiento pertinente.

A medida que el proceso avanza, el estado de va actualizando de forma incremental, acumulando a su vez información relevante que podría influir en decisiones y transiciones futuras.
Este ciclo de ejecución de nodo, actualización de estado y transición se repite iterativamente hasta que se alcanza la condición de parada, previamente establecida. \cite{anderfernandez_langgraph2024}

En aquellos casos en los que el grafo incluye bucles, el sistema puede volver a nodos previamente ejecutados, lo que permite implementar procesos iterativos.
Este tipo de comportamiento resulta especialmente útil en escenarios en los que, antes de obtener un resultado final, es necesario refinar prograsivamente una solución o evaluar
ditintas alternativas.

La ejecución finaliza cuando se cumple una condición de parada \cite{nocedal2006numerical}, obteniéndose un resultado generado a partir del estado acumulado. De esta forma, este enfoque permite
construir sistemas en los que la respuesta final es el resultado de múltiples pasos intermedios, en lugar de una única operación directa.


%────────────────────── Gestión del estado en LangGraph ──────────────────────────
\section{Gestión del estado en LangGraph}

Uno de los elementos clave en el funcionamiento de sistemas basados en grafos es la gestión del estado, entendido como el conjunto de información que se mantiene y se actualiza a lo
largo de la ejecución. En el caso de \emph{LangGraph}, el estado desempeña un papel fundamental, ya que actúa como el medio a través del cual los distintos nodos comparten información
y coordinan su comportamiento.

A diferencia de enfoques más simples, en los que cada operación se ejecuta de forma aislada, el uso de un estado compartido permite construir procesos en los que cada paso
depende de los resultados obtenidos previamente. De este modo, el sistema es capaz de mantener un contexto acumulativo, lo que resulta especialmente útil en aplicaciones que requieren
múltiples etapas de procesamiento.

El estado puede contener distintos tipos de información, dependiendo del diseño del sistema. Entre los elementos más habituales se encuentran el historial de interacciones y los resultados
intermedios generados durante la ejecución, así como métricas o puntuaciones asociadas a diferentes alternativas. Esta flexibilidad permite adaptar el estado a las necesidades específicas de
cada aplicación.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.7\textwidth]{images/EstadoMemoria.png}
    \caption{Esquema del estado del sistema que incluye historial, resultados y puntuaciones. Fuente: elaboración propia}
    \label{fig:estado}
\end{figure}

Durante la ejecución del grafo, cada nodo tiene la capacidad de acceder al estado, modificarlo y añadir nueva información. Este proceso de actualización progresiva es lo que permite que el
sistema evolucione a lo largo del tiempo, incorporando nuevos datos que influyen en las decisiones posteriores.

No obstante, una gestión inadecuada del estado puede introducir problemas en el sistema. Por ejemplo, un crecimiento excesivo de la información almacenada puede afectar al rendimiento, mientras
que la inclusión de datos irrelevantes puede dificultar la toma de decisiones. Por ello, resulta necesario diseñar cuidadosamente qué información se almacena y cómo se actualiza en cada etapa
del proceso. \cite{weng2023llm_agents}

En conclusión, la gestión del estado no solo constituye un elemento técnico dentro de \emph{LangGraph}, sino que también representa un aspecto clave en el diseño de agentes basados en modelos
de lenguaje. Un estado bien defeinido permite mejorar la coherencia del sistema, facilitar el seguimiento del razonamiento y optimizar el proceso de toma de decisiones.


%────────────────────── Máquina de estados finitos aplicada a LangGraph ──────────────────────────
\section{Máquina de estados finitos aplicada a LangGraph}
Las máquinas de estados finitos constituyen un modelo ampliamente utilizado para describir sistemas cuyo comportamiento puede dividirse en un conjunto de etapas bien definidas, entre las que se
producen transiciones en función de determinadas condiciones. Este enfoque resulta especialmente útil para modelar procesos en los que el flujo de ejecución no es estrictamente lineal, sino
que depende de decisiones tomadas durante la ejecución. \cite{hopcroft2001automata}

En el contexto de \emph{LangGraph}, este modelo permite interpretar el flujo del agente como una secuencia de estados conectados mediante transiciones. Cada nodo del grafo representa una etapa concreta
del proceso, mientras que las aristas determinan las posibles evoluciones del sistema en función de las condiciones evaluadas.

Como se indica en reiteradas ocasiones de este proyecto, a diferencia de los enfoques secuenciales tradicionales, el uso de una máquina de estados permite definir comportamientos más complejos, como
la posibilidad de reintentar una operación en caso de fallo, retroceder a una etapa anterior o seguir distintos caminos en función del resultado obtenido en un paso intermedio. Este tipo de control
resulta difícil de implementar mediante estructuras lineales, pero se adapta de forma natural a un modelo basado en estados.

En relación con este trabajo, este enfoque resulta especialmente relevante para estructurar el proceso de razonamiento del agente. En lugar de considerar el sistema como una única secuencia de pasos,
es posible dividirlo en distintas fases, como la generación de una solución, su evaluación y la toma de decisiones posteriores. Las transiciones entre estas fases permiten determinar si el proceso debe
continuar, repetirse o finalizar. \cite{harel1987statecharts}

Este modelo aporta, por tanto, una forma clara de organizar el flujo de ejecución, facilitando la implementación de comportamientos dinámicos y adaptativos. Además, permite representar de manera explícita
las posibles rutas de ejecución, lo que contribuye a una mejor comprensión del sistema y a una mayor facilidad para su análisis y depuración.


%────────────────────── Ejemplo de patrón ReAct en LangGraph ──────────────────────────
\section{Ejemplo de patrón ReAct en LangGraph}

Una de las formas más representativas de utilizar \emph{LangGraph} en el diseño de agentes es mediante la implementación de patrones que combinan razonamiento y acción de manera iterativa. Este tipo de
enfoque permite que el sistema no solo genere una respuesta, sino que también interactúe con su entorno y utilice información externa para mejorar sus decisiones. \cite{yao2023react}

En este sentido, resulta especialmente ilustrativo considerar un esquema en el que el agente analiza una consulta inicial y decide si es necesario realizar alguna acción adicional antes de generar una
respuesta definitivo. Este proceso puede modelarse mediante un grafo en el que cada nodo representa una fase diferencia del comportamiento del agente.

En una primera etapa el sistema procesa la entrada del usuario y genera una interpretación inicial del problema. A partir de esta información, el agente puede decidir si dispone de suficiente conocimiento
para responder directamente o si, por el contrario, necesita apoyarse en alguna herramienta externa, como una función de Python o una consulta a una API.

Esta decisión se traduce en una transición condicional dentro del grafo. Si el agente determina que no es necesaria ninguna acción adicional, el flujo de ejecución finaliza y se devuelve una respuesta. En
caso contrario, el sistema transita hacia un nodo encargado de ejecutar la acción correspondiente, obteniendo nueva información que se incorpora al estado.

Una vez se realizada la acción, el flujo no concluye, sino que vuelve al nodo inicial de razonamiento. De este modo, el agente puede reconsiderar la situación a la luz de los nuevos datos obtenidos, generando
un proceso iteerativo en el que se alternan fases de análisis y acción hasta alcanzar una solución adecuada.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.7\textwidth]{images/CicloRazonamientoAcción.png}
    \caption{Esquema de interacción iterativa entre las fases de razonamiento y acción en un agente basado en grafos. Fuente: elaboración propia}
    \label{fig:estado}
\end{figure}

Este tipo de patrón resulta especialmenente útil en problemas donde la información es insuficiente o donde es necesario realizar operaciones intermedias antes de obtener una respuesta final. Además, permite
estructurar el comportamiento del agente de forma clara, separando las fases de razonamiento y ejecución, lo que facilita tanto su comprensión como su implementación.

En el contexto de este trabajo, este enfoque sirve como ejemplo de cómo \emph{LangGraph} permite integrar distintos componentes dentro de un mismo flujo de ejecución, combinando generación, evaluación y toma
de decisiones en un proceso iterativo.

\section{Reference implementation (PyTorch)}
\begin{lstlisting}[style=python,caption={Minimal SR+U\_Net skeleton (PyTorch)},label={lst:pytorch}]
# Module: sr_seg.py
# Purpose: Joint SR reconstruction and segmentation for MRI volumes
# Notes: Typed, logged, dataclass configs, minimal structure

from dataclasses import dataclass
from typing import Tuple
import logging, torch, torch.nn as nn
import torch.nn.functional as F

logger = logging.getLogger("srseg")
logger.setLevel(logging.INFO)

@dataclass
class TrainConfig:
    lr: float = 2e-4
    gamma: float = 1.0
    lambda_l1: float = 1.0
    lambda_ssim: float = 0.2
    lambda_tv: float = 1e-5

class ResidualBlock(nn.Module):
    """Residual block with two conv3d layers."""
    def __init__(self, c: int):
        super().__init__()
        self.c1 = nn.Conv3d(c, c, 3, padding=1)
        self.c2 = nn.Conv3d(c, c, 3, padding=1)
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        r = F.relu(self.c1(x))
        r = self.c2(r)
        return F.relu(x + r)

class SRNet(nn.Module):
    """EDSR-like SR backbone for volumetric data."""
    def __init__(self, in_ch: int = 1, base: int = 64, depth: int = 8):
        super().__init__()
        self.head = nn.Conv3d(in_ch, base, 3, padding=1)
        self.body = nn.Sequential(*[ResidualBlock(base) for _ in range(depth)])
        self.tail = nn.Conv3d(base, 1, 3, padding=1)
    def forward(self, y: torch.Tensor) -> torch.Tensor:
        h = self.head(y)
        h = self.body(h)
        return self.tail(h)

class UNet3D(nn.Module):
    """3D U-Net for segmentation."""
    def __init__(self, in_ch: int = 1, n_classes: int = 2, base: int = 32):
        super().__init__()
        self.e1 = nn.Sequential(nn.Conv3d(in_ch, base, 3, padding=1), nn.ReLU(),
                                nn.Conv3d(base, base, 3, padding=1), nn.ReLU())
        self.e2 = nn.Sequential(nn.Conv3d(base, base*2, 3, stride=2, padding=1), nn.ReLU())
        self.b  = nn.Sequential(nn.Conv3d(base*2, base*4, 3, padding=1), nn.ReLU())
        self.d2 = nn.Sequential(nn.ConvTranspose3d(base*4, base*2, 2, stride=2), nn.ReLU())
        self.out= nn.Conv3d(base*2, n_classes, 1)
    def forward(self, x: torch.Tensor) -> torch.Tensor:
        x1 = self.e1(x)
        x2 = self.e2(x1)
        b  = self.b(x2)
        d2 = self.d2(b)
        z  = self.out(d2)
        return z

def dice_loss(p: torch.Tensor, m: torch.Tensor, eps: float = 1e-6) -> torch.Tensor:
    p = p.softmax(1)[:,1]  # foreground
    m = (m > 0.5).float()
    num = 2*(p*m).sum()
    den = p.sum() + m.sum() + eps
    return 1 - num/den

def tv(x: torch.Tensor) -> torch.Tensor:
    return (x[:, :, 1:] - x[:, :, :-1]).abs().mean() + \
           (x[:, :, :, 1:] - x[:, :, :, :-1]).abs().mean() + \
           (x[:, :, :, :, 1:] - x[:, :, :, :, :-1]).abs().mean()

def train_step(sr: SRNet, unet: UNet3D, opt: torch.optim.Optimizer,
               y: torch.Tensor, x: torch.Tensor, m: torch.Tensor,
               cfg: TrainConfig) -> float:
    opt.zero_grad()
    x_hat = sr(y)
    l1 = (x_hat - x).abs().mean()
    # crude SSIM proxy to keep example short
    ssim_term = ((x_hat - x)**2).mean()
    l_tv = tv(x_hat)
    z = unet(x_hat.detach())
    ce = F.cross_entropy(z, m.long())
    dl = dice_loss(z, m)
    loss = cfg.lambda_l1*l1 + cfg.lambda_ssim*ssim_term + cfg.lambda_tv*l_tv + cfg.gamma*(ce + dl)
    loss.backward()
    opt.step()
    logger.info(f"loss={loss.item():.4f}")
    return float(loss.item())
\end{lstlisting}

# eval_metrics.py
# Computes PSNR, SSIM (skimage), and Dice given predictions and references.

from skimage.metrics import structural_similarity as ssim
import numpy as np

def psnr(x, y, peak=1.0):
    mse = np.mean((x - y) ** 2)
    return 10 * np.log10(peak**2 / (mse + 1e-12))

def dice(p, m, thr=0.5):
    pbin = (p >= thr).astype(np.float32)
    m = m.astype(np.float32)
    return 2*(pbin*m).sum() / (pbin.sum() + m.sum() + 1e-12)

// conv3d_kernel.cpp - High-performance 3D convolution
#include <vector>
#include <cmath>
#include <algorithm>

template<typename T>
class Conv3DKernel {
private:
    std::vector<T> weights_;
    int kernel_size_;

public:
    Conv3DKernel(int size) : kernel_size_(size) {
        weights_.resize(size * size * size);
    }

    // Apply convolution at a single point
    T apply(const T* input, int x, int y, int z, int stride) const {
        T sum = static_cast<T>(0);
        for (int dz = 0; dz < kernel_size_; ++dz) {
            for (int dy = 0; dy < kernel_size_; ++dy) {
                for (int dx = 0; dx < kernel_size_; ++dx) {
                    int idx = (z + dz) * stride * stride
                            + (y + dy) * stride + (x + dx);
                    sum += input[idx] * weights_[dz * kernel_size_ * kernel_size_
                                                + dy * kernel_size_ + dx];
                }
            }
        }
        return sum;
    }
};
\end{lstlisting}

-- Select patients with complete MRI sequences for training
SELECT DISTINCT
    p.patient_id,
    p.age,
    p.diagnosis_date,
    COUNT(s.scan_id) AS num_scans
FROM patients p
INNER JOIN mri_scans s ON p.patient_id = s.patient_id
WHERE s.modality = 'T1w'
  AND s.quality_score >= 0.8
  AND p.diagnosis_date BETWEEN '2020-01-01' AND '2024-12-31'
GROUP BY p.patient_id, p.age, p.diagnosis_date
HAVING COUNT(s.scan_id) >= 3
ORDER BY p.diagnosis_date DESC
LIMIT 1000;
